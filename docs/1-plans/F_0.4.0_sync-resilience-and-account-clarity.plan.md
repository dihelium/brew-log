# Sync Resilience & Account Clarity Implementation Plan

## Overview

Harden the outbox sync path and make account/sync state visible, in response to a
confirmed production incident: the user's iPhone accumulated 39 real brews of which
only 22 (by id) are on the server — 17 July entries are missing and the iPhone's own
writes never synced — while Google OAuth silently auto-selected an account with no
chooser. This plan makes `flushOutbox` resilient (one failing op can no longer
strand all later ops), surfaces sync failures with an error chip, forces a Google
account chooser, and shows the signed-in email with a sign-out control. It also
documents a safe procedure to consolidate the backed-up July data onto the
canonical account.

## Problem Statement

Investigation (server-side, via the user's own session) established:

- Supabase holds 22 rows (May 25–Jun 20) for `varshneybhargav@gmail.com`. Those 22
  ids are a strict subset of the iPhone's 39 local entries (through Jul 15), so **17
  July entries are missing from the server**, and the iPhone's *own* writes never
  synced at all — the 22 arrived via a one-time backup import on the Mac, not from the
  iPhone. (No other server account holds the missing entries either.)
- **Root cause A — silent head-of-line blocking**: `flushOutbox` (`src/lib/sync.js:138`)
  iterates queued ops in order and `return`s on the *first* error, leaving that op
  and everything after it in the outbox. A single persistently-failing op (e.g. one
  photo upload) freezes **all** sync, with no user-visible signal.
- **Root cause B — silent account selection**: `signInWithGoogle`
  (`src/context/AuthContext.jsx`) calls `signInWithOAuth` with no `prompt`, so a
  device auto-signs into whatever Google account is default — a user can unknowingly
  log to the wrong account. There is also **no UI anywhere** to see the signed-in
  account or to sign out (a real user cannot currently sign out at all).

The exact iPhone-side failure mechanism (stuck outbox vs. wrong account vs. stale
PWA) could not be isolated — no device access. These are defensive fixes targeting
the plausible causes and, critically, making sync **observable and recoverable** so
the failure is self-diagnosable next time.

## Solution Architecture

Four code changes plus one operational procedure:

1. **Resilient `flushOutbox`** — process every queued op, skipping (not aborting on)
   failures, while **preserving per-entry causal order**: if an op for entry X fails,
   block only entry X's later ops this pass; ops for other entries still flush.
   Failed ops stay queued and retry on the next pass. Return a summary so callers know
   a failure occurred. (User decision: skip-and-continue, **no** parking/retry-cap.)

2. **Sync-error visibility** — `BrewContext` tracks a `syncError` flag set when a
   flush leaves failures (or `pullRemote` errors) and cleared on a fully clean pass.
   `FeedPage` shows a minimal "Sync failed — tap to retry" chip only when it is set.
   (User decision: error-only chip, not a persistent indicator.)

3. **Forced account chooser** — add `queryParams: { prompt: 'select_account' }` to the
   Google OAuth call so the account picker always appears.

4. **Account visibility + sign-out** — a small "Signed in as \<email\>" row with a
   Sign out button on `FeedPage` (real sessions only; demo keeps its existing
   `DemoBanner`).

5. **Consolidation procedure** (operational, not code) — bring the 39 backed-up
   entries onto `varshneybhargav@gmail.com`, honoring the user's "fix iPhone first"
   preference, with a guaranteed Mac-import fallback (see Backward Compatibility).

No entry-schema change, no `toRow`/`fromRow` change, no IndexedDB version bump, no
Supabase migration.

## Implementation Details

### 1. Resilient `flushOutbox`

**File**: `src/lib/sync.js`

**Current state** (lines 138–178): loops `ops` in seq order; each op's push is wrapped
in `try/catch`; on catch it `return { ok: false, error }`, abandoning the pass and
leaving the failing op **and every later op** queued. On success it `removeOp`s.

**Modifications**:

- Replace the early `return` on error with **skip-and-continue**:
  - Maintain a `blockedIds` set (entry ids whose earlier op failed this pass).
  - For each op, resolve its entry id (all three op shapes carry `item.entry.id`).
    - If `blockedIds` has that id → **skip** the op (leave queued), do not attempt it
      (preserves causal order: never apply an `update`/`delete` ahead of a failed
      `add` for the same entry).
    - Else attempt the push (add/update/delete — see the idempotent-upload and
      delete-`remove()` modifications below, which change the add and delete bodies).
      On success → `removeOp`. On failure → add id to `blockedIds`, record the first
      error, leave the op queued, continue.
- Change the return shape to a summary: `{ ok, flushed, failed, cleanupFailed, error }`
  where `ok === (failed === 0 && cleanupFailed === 0)`, `flushed` = count removed,
  `failed` = count of distinct blocked ops/entries, `cleanupFailed` = count of
  non-blocking storage-cleanup failures (delete `remove()`), `error` = first error
  encountered (or null). Additive; existing callers that ignored the return still work.
- Ordering guarantee preserved: ops are still visited in seq order; only the
  *failure* behavior changes from "abort all" to "skip this entry's chain."

**Idempotent photo upload (retry-safety — Codex P1)**: the current `add` path uploads
with `{ upsert: true }` (line 149). If the photo upload *succeeds* but the row upsert
*fails* (transient), the op stays queued and retries; on retry the object already
exists, and if the Storage bucket lacks an owner `UPDATE` policy an `upsert:true`
overwrite is rejected — permanently stranding that entry. This exact partial-failure
is a plausible contributor to the iPhone incident. Fix so retry needs only the
`INSERT` policy: attempt the upload and **treat an "already exists"/duplicate object
result as success** (i.e. the photo from a prior attempt is already there), rather
than depending on overwrite. Concretely, switch to `{ upsert: false }` and, when the
upload returns a duplicate/conflict error (Supabase `409` / "The resource already
exists"), continue as if it succeeded; only a genuine upload error blocks the entry.
This makes the add path idempotent under retry without requiring an `UPDATE` policy.

**Storage errors on delete/pull (Codex P2)**:
- `delete` op currently ignores `storage.remove()`'s returned error (line 169). Keep
  the row delete authoritative (the entry *is* removed server-side — the user's
  intent), but **read** the `remove()` result: log a failure and increment a separate
  `cleanupFailed` counter in the return contract (see Type Definitions) so a leaked
  object is visible via `syncError`. Do **not** gate op removal on it — gating would
  create a permanent delete-retry loop (row already gone) that pins the chip; the
  one-time `cleanupFailed` signal is enough and clears on the next clean pass.
- `pullRemote` currently ignores `storage.download()`'s error (line 220). Capture it:
  a failed photo download should mark the pass as not-fully-clean (feeds `syncError`)
  so the chip reflects that a photo didn't arrive, while the entry row still caches.

**Why not parking/retry-cap**: user chose plain skip-and-continue. A permanently-bad
op will retry each pass; the error chip (below) makes that visible and the user can
decide. Parking is deferred (noted as possible future work, not implemented).

### 2. Sync-error state in `BrewContext`

**File**: `src/context/BrewContext.jsx`

**Current state**: `runSync` (lines 59–81) calls `flushOutbox` then `pullRemote` then
`hydrate`, ignoring their return values; no error state exists.

**Modifications**:

- Add `syncError` state (boolean), default `false`.
- In `runSync`, capture `flushOutbox`'s summary and `pullRemote`'s result. `pullRemote`
  gains a soft-failure signal for photo-download errors (e.g. `{ ok, count, photoFailed }`)
  so a partial pull is distinguishable from a query failure. After a pass: if
  `flushResult.failed > 0 || flushResult.cleanupFailed > 0`, `pullRemote` returned
  `ok:false`, `photoFailed` is set, or either threw → `setSyncError(true)`; on a fully
  clean pass → `setSyncError(false)`. Keep the existing queued-follow-up loop intact.
- Demo mode is unaffected: `runSync` already early-returns when `isDemo`, so
  `syncError` stays `false` in demo.
- Expose `syncError` and a `retrySync` (thin wrapper calling `runSync`) through the
  context value, alongside the existing `demoSeedError`.

### 3. Sync-error chip on `FeedPage`

**File**: `src/pages/FeedPage.jsx` + `src/components/SyncErrorChip.jsx` (new)

- New `SyncErrorChip`: a slim, theme-variable-styled chip ("⚠️ Sync failed — tap to
  retry") that calls `retrySync` on tap. Non-fixed; flows above the heading like
  `DemoBanner`. Uses only theme CSS variables, ≥44px touch target.
- `FeedPage` renders `<SyncErrorChip />` when `syncError` is true and `!isDemo`.

### 4. Forced account chooser

**File**: `src/context/AuthContext.jsx`

**Current state** (`signInWithGoogle`, lines 43–49): `signInWithOAuth({ provider:
'google', options: { redirectTo: window.location.origin } })`.

**Modifications**: add `queryParams: { prompt: 'select_account' }` to `options` so
Google always shows the account picker. One-line, behavior-only change.

### 5. Account visibility (extend existing `BackupControls`) + import bugfix

**File**: `src/components/BackupControls.jsx` (modify)

**Current state**: `BackupControls` already renders a **Sign out** button (lines
122–139) and owns the account-adjacent controls. It also has a latent bug: `handleImportFile`
calls `const { added, valid } = importEntries(raw)` **without `await`** (line 60) —
`importEntries` is async, so `added`/`valid` are `undefined` and the status message is
always wrong ("Restored undefined brews"), even though the import itself runs. This
matters because the Mac-import consolidation fallback relies on this control.

**Modifications**:
- **Do not** add a second sign-out (Codex P2): instead add a "Signed in as
  \<user.email\>" line to `BackupControls`, above its existing Sign out button. Pull
  `user` from `useAuth()` (already imported there). Theme-styled, real sessions only.
- **Fix the import** (Codex P2): `await importEntries(raw)` so `added`/`valid` are real
  and the status message ("Restored N brews" / "Already up to date") is correct — the
  fallback needs a trustworthy result.
- No new `AccountControls` component is created (avoids the duplicate sign-out).

## Technical Considerations

- **Pattern Usage**: `flushOutbox` stays a pure async function over the cache +
  supabase client (node-testable). The chip/controls follow the existing `DemoBanner`
  component pattern and theme-variable convention.
- **Offline-First Impact**: no entry-schema change, no new fields, `toRow`/`fromRow`
  untouched, no `isValidEntry` change. Cached entries load exactly as before.
- **Sync Safety** (critical path): outbox **ordering is preserved** — ops still
  processed in seq order; the only change is failure handling (skip the failing
  entry's chain vs. abort the whole pass). Per-entry causal order (add→update→delete)
  is explicitly protected via `blockedIds`. Per-user cache isolation
  (`brew-log-<userId>`) is unchanged. `delete-wins` and pending-patch-overlay
  semantics in `pullRemote` are untouched.
- **Mobile/PWA UX**: chip and account row are ≥44px targets, theme-styled, safe-area
  friendly, and work offline (chip is the offline/failure signal). No service-worker
  change.
- **Theming**: all new UI uses theme CSS variables (checked against all five themes);
  no hardcoded colors.
- **Performance**: no new re-render storms — `syncError` toggles at most once per sync
  pass; account row reads a stable `user.email`.
- **Edge Cases**: clean pass clears the chip; demo mode never shows it; a persistently
  failing op keeps the chip lit (intended, user-visible); signing out returns to
  `LoginPage`; forced chooser does not affect already-valid sessions (only new
  sign-ins).

## Type Definitions

- `flushOutbox` return: `{ ok, flushed, failed, cleanupFailed, error }` where
  `failed` counts entries blocked by a push failure, `cleanupFailed` counts
  non-blocking storage-cleanup failures (delete `remove()`), and
  `ok === (failed === 0 && cleanupFailed === 0)`. Was `{ ok, error? }` — additive/
  back-compatible for callers reading `ok`.
- `runSync` sets `syncError` when `failed > 0 || cleanupFailed > 0`, `pullRemote`
  returned `ok:false` or `photoFailed`, or either threw.
- `pullRemote` return gains an optional `photoFailed: boolean` soft-failure signal
  (existing `{ ok, count }` unchanged otherwise).
- No persisted-type or entry-shape changes.

## Performance & Cost Impact

- Negligible. Resilient flush may attempt more ops per pass (other entries no longer
  blocked), which is the desired behavior; each op is the same Supabase call as today.
- No extra reads/writes for real users beyond what a healthy flush already does.

### Consolidation procedure (operational — the 39 backed-up entries)

Canonical account: **`varshneybhargav@gmail.com`**. The July backup
(`brew-log-backup-2026-07-15.json`, 39 entries, verified) is the safety net, so no
step below can lose data.

**Important caveat (per-user DB isolation, ARCHI §5)**: the iPhone's 39 entries live
in `brew-log-<the-account-the-iPhone-used>`. Signing the iPhone into a *different*
account opens a *different* (empty) local DB — the 39 entries only auto-flush if the
iPhone is already on the canonical account.

Procedure (honors "fix iPhone first", with a guaranteed fallback):

1. Ship this release; deploy to production.
2. On the iPhone, update the installed PWA to the new build (may require closing/
   reopening, or removing & re-adding the home-screen app to bust the cached service
   worker).
3. **Read the account BEFORE any sign-out** (Codex P1): the new "Signed in as …" row
   shows the iPhone's *current* account without touching the session — do not sign out
   first, or you lose the evidence about where the 39 entries live.
   - **If it shows `varshneybhargav@gmail.com`**: the 39 entries are in the canonical
     account's local DB. The resilient `flushOutbox` pushes them up; the error chip
     flags any op that still fails. No account switch needed. ✅
   - **If it shows a different account**: those 39 entries belong to *that* account's
     local DB, and signing into `varshneybhargav` would open a different, empty DB
     (per-user isolation). So **do the Mac-import fallback first**, before switching:
     import `brew-log-backup-2026-07-15.json` on the **Mac** (signed into
     `varshneybhargav`, whose sync is proven) via the fixed `BackupControls` import —
     dedups by id and pushes all 39 to the server under the canonical account. Only
     then switch the iPhone to `varshneybhargav` (the forced chooser makes this
     reliable) for future logging.
4. **Verify server persistence, not local cache** (Codex P2): the Mac feed reads the
   local cache and cannot prove the server has the rows. Confirm via the Supabase
   dashboard SQL editor (the same `entries` ⋈ `auth.users` grouping query used during
   diagnosis). **Exact expected count: 39 rows** for `varshneybhargav`, newest July 15.
   Verified basis: the July backup's 39 ids are a strict superset of the server's
   current 22 (all 22 June ids ∈ the 39 backup ids; 17 are new), so importing dedups by
   id and the union is exactly 39 — not 61. (If a future re-check shows a different
   number, compare the backup ids against the server ids directly.)

This procedure is documentation + user action after deploy; it introduces no code
beyond items 1–6.

## Test Impact

- **`src/lib/sync.js` (new behavior, node-testable via existing `memCache` +
  supabase mock in `src/lib/sync.test.js`)**:
  - A failing op for entry A does **not** block entry B: B flushes (removed from
    outbox), A's op stays queued; return summary reports `failed >= 1`.
  - Per-entry ordering: when an `add` for entry A fails, a later `update` for A is
    **not** attempted this pass (stays queued), while another entry still flushes.
  - **Idempotent photo add (Codex P1)**: simulate upload-success → row-upsert-failure,
    then a retry pass where the object "already exists" → the retry treats the
    duplicate as success and the entry finally flushes (op removed). This pins the
    retry-safety fix.
  - **Delete cleanup signal**: a `delete` whose `storage.remove()` errors still removes
    the op (row delete authoritative) but returns `cleanupFailed >= 1` (→ `syncError`).
  - All-success pass: every op removed, `ok:true, failed:0, cleanupFailed:0`.
  - Extend the mock with failing variants (error on chosen ids, on `storage.upload`,
    on `storage.remove`, and a duplicate/409 upload result) — small additions alongside
    the existing `okSupabase` factory.
- **Not unit-tested (manual/browser)**: the `syncError` chip wiring, the "Signed in as
  \<email\>" row in `BackupControls`, and the `prompt: 'select_account'` OAuth param
  live in the React/OAuth layer (no harness per ARCHI §10). Verify manually: force a
  flush failure (e.g. offline) → chip appears → reconnect + tap retry → chip clears;
  sign-in shows the account chooser; the account row shows the correct email and sign-
  out returns to login; an import reports the correct restored-count (await fix).
