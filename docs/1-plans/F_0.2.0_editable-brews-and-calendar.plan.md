# Editable Brews + Calendar View Implementation Plan

## Overview

Three related additions to my-brew-log:

1. **Editable fields** — make an existing brew's **date & time, rating, notes, type, and name** editable inline on the Detail page (matching the existing tap-to-edit location pattern).
2. **New "tea" brew type** — add `tea` alongside `coffee` and `matcha` everywhere a type is chosen, rendered, or themed.
3. **Calendar view** (non-default route `/calendar`) — a month grid with a dot on each day that has brews; selecting a day shows that day's brew list plus a "daily cup" mug rendered from that day's colors, i.e. what the mug looked like on that day.

All three ship together as **v0.2.0** (minor feature release).

## Problem Statement

- Once a brew is logged, only its **location** can be changed. A wrong date, rating, note, type, or name is permanent unless the entry is deleted and re-added (which loses its photo/color/streak position).
- Only `coffee` and `matcha` types exist; the user also drinks tea and cannot classify it.
- Brews can only be viewed as a reverse-chronological feed and a "today" cup. There is no way to jump to a past date, see that day's brews, or see the cup for that day.

## Solution Architecture

- **Generalize the existing update/outbox patch path** (`updateEntry` in `BrewContext`, `toPatchRow`/`applyPatch`/`normalizePatch` in `sync.js`) from location-only to a small fixed set of patchable client fields: `name`, `type`, `rating`, `notes`, `timestamp`, `location`. This reuses the proven outbox `update` op, mid-sync overlay, and per-user cache isolation with **no new IndexedDB store, no DB version bump, and no Supabase migration** (all target columns already exist in `toRow`).
- **Centralize brew-type metadata** (id, label, emoji) in a new pure, node-testable `src/utils/brewTypes.js`, and drive `AddPage`, `EntryCard`, `DetailPage` from it. Add `tea` there once and add its theme CSS variables + `data-type` rules.
- **Add a `/calendar` route** rendering a new `CalendarPage`. Month-grid math and per-day filtering live in a new pure `src/utils/calendar.js`. The mug is the **existing `DailyCup` component**, extended with optional caption props so its "today" wording can be made date-neutral without breaking the Feed page.

### 5. Generalize the update / outbox patch path

**File**: `src/lib/sync.js`

**Current state**: `toPatchRow` (lines 44–50) and `applyPatch` (lines 56–63) handle **only** `location`. The outbox `update` op flush (lines 86–91) calls `.update(toPatchRow(patch)).eq('id', ...)`. `pullRemote` overlays merged pending patches via `applyPatch` (lines 132–136, 150).

**Modifications** — support a fixed patch key set: `name`, `type`, `rating`, `notes`, `timestamp`, `location`. Keep everything symmetric and backward compatible.

- Add `normalizePatch(patch)` (new exported pure function) that produces a canonical, client-shaped patch:
  - `name`: if present and `String(patch.name).trim()` is non-empty → `trimmed`; if present but empty → **omit the key** (name is required and must never be cleared).
  - `type`: if present → the value as-is (one of the `BREW_TYPES` ids, or `null`).
  - `rating`: if present → a number `1..5`, or `null` when falsy/`0` (clears).
  - `notes`: if present → trimmed string, or `null` when empty (clears).
  - `location`: if present → trimmed string, or `null` when empty (clears) — preserves today's behavior.
  - `timestamp`: if present → a finite number (ms epoch); omit if not finite.
  - Unknown keys are dropped.
- Extend `toPatchRow(patch)` to map each present canonical key to its column:
  - `name → row.name` (string)
  - `type → row.type` (value or `null`)
  - `rating → row.rating` (`patch.rating ?? null`)
  - `notes → row.notes` (`patch.notes || null`)
  - `location → row.location` (`patch.location || null`) — unchanged behavior
  - `timestamp → row.logged_at = new Date(patch.timestamp).toISOString()`
  - Uses `Object.prototype.hasOwnProperty` guards per key (same style as today) so absent keys never touch a column.
- Extend `applyPatch(entry, patch)` to mirror `toPatchRow` on the client shape:
  - `name` present → `next.name = patch.name`
  - `type` present → `next.type = patch.type ?? null`
  - `rating` present → set `next.rating` when truthy, else `delete next.rating`
  - `notes` present → set when truthy, else `delete next.notes`
  - `location` present → existing set/delete behavior
  - `timestamp` present → `next.timestamp = patch.timestamp`
  - (Delete-on-empty for optional fields keeps cache entries schema-consistent with add path, where cleared optional fields are absent.)
- `flushOutbox` update branch is unchanged in shape — it already calls `toPatchRow(item.entry.patch)`; the widened mapping flows through automatically. Update ops still never touch photo storage or `photo_path`.
- `pullRemote` overlay is unchanged in shape — `applyPatch(fromRow(row), mergedPatch)` now overlays the new keys too, so an edit enqueued during a sync pass is not reverted by the stale remote row.

**Sync-safety notes**:
- Outbox ordering and mid-sync follow-up-pass (`runSync`) are untouched; per-user cache DB name is untouched; no DB version bump (only the existing `entries`/`photos`/`outbox` stores).
- **Retry safety, stated correctly**: the `update` op is `.update(cols).eq('id', id)` — **not an upsert**. It is *idempotent* (re-applying the same patch is safe on retry), which is what matters for outbox replay.
- **Remotely-deleted-row conflict (accepted behavior, unchanged from today's location edit)**: if the target row was deleted on another device, `.update().eq()` succeeds matching **zero** rows, the outbox op is removed, and the next `pullRemote` prunes the (now server-absent) local entry. This is the project's existing **delete-wins** conflict outcome; this plan does **not** change it and deliberately does **not** resurrect the row via upsert (that would let a stale edit silently override a concurrent delete). Add a `sync.test.js` case asserting an `update` to a server-absent id is dropped and the local entry is pruned by the following pull, so the behavior is pinned rather than accidental. Note this in `docs/ARCHI.md` §9 during TRIP-3 if it isn't already explicit.

**File**: `src/context/BrewContext.jsx`

**Current state**: `updateEntry(id, patch)` (lines 147–167) hard-guards to `patch.location` only and hand-rolls the location set/delete.

**Modifications**:
- Import `applyPatch`, `normalizePatch` from `../lib/sync`.
- Rewrite `updateEntry` generically:
  1. Guard `cache` exists.
  2. `const normalized = normalizePatch(patch)`; if it has **zero keys**, return (nothing to do — e.g. an attempt to blank the name).
  3. `const existing = await cache.getEntry(id)`; if missing, return.
  4. `const nextEntry = applyPatch(existing, normalized)`.
  5. `await cache.putEntry(nextEntry)`.
  6. `await cache.enqueue('update', { id, patch: normalized })`.
  7. `setEntries(prev => prev.map(e => e.id === id ? applyPatch(e, normalized) : e).sort((a, b) => b.timestamp - a.timestamp))` — `applyPatch` spreads the entry so the attached `photo` object URL is preserved, and the **re-sort keeps the feed in descending-timestamp order after a date edit** (matching `hydrate`'s sort; without it a re-dated card would keep its old position until the next hydrate, especially while offline).
  8. `runSync()`.
- This keeps the current location edit working unchanged (callers still pass `{ location }`), and the hydrate/URL logic is untouched (edits never change `hasPhoto`).

### 6. Inline per-field editing on the Detail page

**File**: `src/pages/DetailPage.jsx`

**Current state**: location has a bespoke inline editor (`editingLocation` boolean + `locationDraft`, lines 20–61, 103–136). Date is display-only (`formatDate`, line 100). Rating is read-only `StarRating` (line 140). Notes is display-only (line 145). Name + type live in the hero (lines 83–86).

**Modifications** — unify editing under a single `editingField` state and add editors for each field, following the existing location tap→edit→Save/Cancel pattern. Keep location behavior identical.

- Replace `editingLocation` boolean with `const [editingField, setEditingField] = useState(null)` where the value is one of `null | 'header' | 'date' | 'rating' | 'notes' | 'location'`. Only one field edits at a time.
- Draft state per field: `nameDraft`, `typeDraft` (for the `header` editor), `dateDraft` (datetime-local string), `ratingDraft` (number), `notesDraft` (string), and the existing `locationDraft`.
- Each `begin<Field>Edit()` seeds its draft(s) from `entry` and sets `editingField`. Each `handle<Field>Save()` calls `await updateEntry(id, { <field>: draft })` then `setEditingField(null)`. Cancel just clears `editingField`.

Field editors:
- **Header (name + type)** — the hero name (`detail-hero__name`) and badge become a single tappable affordance (`type="button"`, `aria-label="Edit name and type"`). When `editingField === 'header'`, render an inline editor **in the detail body** (below the date, above location so controls are on the readable surface, not over the photo): a text `<input>` bound to `nameDraft` and a three-button type toggle (reuse the `sheet__type-btn` style, driven by `BREW_TYPES`) bound to `typeDraft`, plus Save/Cancel. Save calls `updateEntry(id, { name: nameDraft, type: typeDraft })`. If `nameDraft` trims empty, disable Save (name required — matches `normalizePatch` dropping empty name).
- **Date & time** — the existing date line (line 100) becomes a tappable button. Editor: `<input type="datetime-local">` bound to `dateDraft`, Save/Cancel. Convert with the new `datetimeLocal` util (section 7): seed `dateDraft = toDatetimeLocalValue(entry.timestamp)`; on Save `updateEntry(id, { timestamp: fromDatetimeLocalValue(dateDraft) })`. If the parsed value is not finite, disable Save.
- **Rating** — the rating area becomes tappable (including when currently unrated, so a rating can be added). Editor: an editable `StarRating` bound to `ratingDraft` (`onChange={setRatingDraft}`), a **"Clear" text button**, and Save/Cancel. Because the existing `StarRating` only selects `1..5` and has no zero state, clearing is handled **in DetailPage** (not by modifying `StarRating`): the Clear button sets `ratingDraft = 0`, and Save always sends `updateEntry(id, { rating: ratingDraft })` (`0` → normalized to `null` → column cleared). `StarRating` renders `ratingDraft = 0` as all-empty stars (its `value = 0` default already does this), so no change to that component is needed. Display when not editing: keep read-only stars if `entry.rating > 0`, otherwise show a subtle "Add rating" affordance (mirroring the empty-location row).
- **Notes** — notes area becomes tappable (including when empty → "Add notes"). Editor: `<textarea>` bound to `notesDraft`, Save/Cancel; empty clears.
- **Location** — unchanged behavior, migrated to the shared `editingField` state.

Styling: reuse existing classes where possible (`detail-location-edit`, `detail-location-actions`, `detail-location-save/cancel`, `sheet__input`, `sheet__textarea`, `sheet__type-btn`). Add a few new classes in `index.css` for the editable rows/buttons (e.g. `detail-edit-row`, `detail-edit-actions`) using theme variables only. Touch targets ≥ 40px tall for mobile; respect existing spacing.

**Edge cases**: entry not found (existing guard stays); editing while offline just enqueues the outbox `update` op (works from cache); rapid consecutive edits each enqueue an update and are merged by `pullRemote`'s per-id patch merge.

### 7. Datetime-local conversion util

**File**: `src/utils/datetimeLocal.js` (new)

Pure, node-testable. Handles the `<input type="datetime-local">` ↔ ms-epoch conversion in **local** time (the input has no timezone):

- `toDatetimeLocalValue(ts)` → `'YYYY-MM-DDTHH:mm'` built from a local `Date` (zero-padded); returns `''` for non-finite input.
- `fromDatetimeLocalValue(str)` → ms epoch via `new Date(str).getTime()` (browser parses `datetime-local` as local time); returns `NaN` for empty/invalid, so callers can guard.

### 8. Calendar month-grid + filtering util

**File**: `src/utils/calendar.js` (new)

Pure, node-testable. No DOM. Uses local-time `Date` semantics to match how the Feed groups "today" (`toDateString()`).

- `dayKey(ts)` → a stable local day key for grouping, e.g. `new Date(ts).toDateString()` (reused for both entries and calendar cells so grouping is consistent with FeedPage).
- `buildMonthMatrix(year, month)` → array of weeks, each a 7-length array of cell objects `{ date: Date | null, inMonth: boolean }` (leading/trailing blanks as `null`) covering the weeks that contain `month`. Week starts on Sunday to match the `S M T W T F S` header.
- `brewDaysInMonth(entries, year, month)` → `Set<number>` of day-of-month numbers that have ≥1 entry in that month/year (for dot markers).
- `entriesForDay(entries, date)` → entries whose `dayKey` matches `date`, **sorted ascending by timestamp** (so `DailyCup` bands read oldest→newest bottom→top, same as Feed's `todayEntries`).

## Technical Considerations

- **Pattern Usage**: reuses the outbox `update` op + `toPatchRow`/`applyPatch` symmetry, the tap-to-edit location UX, the `DailyCup` component, and the Feed's day-grouping (`toDateString`). New logic goes into pure `src/utils/*` modules per conventions.
- **Offline-First Impact**: no entry-schema field is added (schema is unchanged; `tea` is just a new value of the existing optional `type`). `isValidEntry` is unaffected (`name`/`timestamp` still required). Cached entries without a rating/notes remain valid. Edits work fully from cache while offline.
- **Sync Safety**: `toRow`/`fromRow` unchanged and still symmetric; `toPatchRow`/`applyPatch` extended symmetrically. Outbox ordering, idempotent `update`-by-id operations (`.update().eq()`, not upserts), retry-on-failure, mid-sync follow-up pass, and per-user `brew-log-<userId>` isolation all preserved. **No IndexedDB version bump, no Supabase migration** (target columns already exist).
- **Mobile/PWA UX**: all new controls ≥ 40px touch targets; calendar grid fits narrow screens and respects safe areas; works offline; no new network dependency. Editing is inline, no route change.
- **Theming**: `tea` accents added to all five theme blocks in `index.css`; every new component/element uses CSS variables — no hardcoded colors in JSX.
- **Performance**: no new photo work; edits don't touch photo storage or object URLs. Calendar util is O(entries) per render for the visible month; acceptable at expected volumes. `updateEntry` updates state via a single `map`, no re-render storm.
- **Edge Cases**: empty/absent rating and notes (add vs edit vs clear); name cannot be blanked; invalid/empty datetime-local guarded; days with no brews (empty mug + empty list); editing while signed out is impossible (no cache) — `updateEntry` early-returns; unknown/legacy `type` values render via `brewTypeLabel` fallback.
