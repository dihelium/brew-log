# Service Worker: Stop Caching Supabase Reads Implementation Plan

## Overview

Fix a confirmed production bug where the service worker serves **stale Supabase
API reads**, so no device ever sees new server data. `public/sw.js` uses a
cache-first strategy for all non-navigate GETs — including cross-origin Supabase
REST/Storage calls — so it cached the `/rest/v1/entries` response when the server
had 22 rows and has served that stale response ever since (the DB has 40). The
fix makes the service worker bypass its cache for Supabase requests (always
network), while keeping cache-first for same-origin app assets and Google Fonts.

## Problem Statement

Confirmed via device diagnostics: on a device signed into the correct account
(`varshneybhargav`), the app's `GET /rest/v1/entries` returns **status 200 with
22 rows, newest June 20**, with no errors — while the database holds **40 rows**.
Ticking DevTools → "Bypass for network" and reloading immediately showed all 40.

- **Root cause**: `public/sw.js`'s `fetch` handler applies a **cache-first**
  strategy (`caches.match` → return cached if present) to *every* non-navigate
  GET, with no origin check. The cross-origin Supabase REST response was cached
  on first success (22 rows) and is returned from cache forever; the network is
  never consulted again for that URL. Writes are `POST`/`PATCH` (the handler
  `return`s early for non-GET), so mutations reached the server normally — which
  is exactly why data got *to* the server (40 rows) but no device could *read*
  the update.
- **Pre-existing**: `sw.js` was not touched by v0.3.0 or v0.4.0; the bug only
  became visible once the server row count actually changed (22 → 40).

## Solution Architecture

Changes to `public/sw.js` (bypass + cache-version bump to purge the poisoned
cache) plus a small `src/main.jsx` addition for correct rollout:

1. **Bypass the SW cache for Supabase requests** — in the `fetch` handler, if the
   request URL's host is a Supabase host (`*.supabase.co`, covering both
   `<ref>.supabase.co/rest/v1/...` and `.../storage/v1/...`), `return` without
   calling `respondWith`, so the browser performs a normal network fetch and the
   SW never reads or writes its cache for that request. Supabase REST responses
   carry no-cache headers, so reads are always fresh.
2. **Keep everything else unchanged** — same-origin navigation stays network-first
   (fallback to cached `index.html`); same-origin static assets and cross-origin
   Google Fonts stay cache-first (so offline app-start and offline theme fonts
   keep working). This honors the "bypass Supabase only" decision.
3. **Bump `CACHE_NAME`** (`brew-log-v1` → `brew-log-v2`) so the existing `activate`
   handler deletes the old cache — purging the stale Supabase responses already
   stored in it — and forces the updated SW to install/activate on devices.
4. **Resync on SW takeover** (`src/main.jsx`) — a guarded `controllerchange`
   listener triggers one resync (via the existing wake event) when the new SW takes
   control, so a fresh pull runs under v2 instead of leaving the user on a stale
   first paint — without a page reload that would drop in-progress drafts (see §2).

Offline behavior is preserved because entry rows and photos are served from
**IndexedDB** (`pullRemote` caches photo blobs), not from the SW HTTP cache; the
SW cache only ever needed to cover the app shell.

## Implementation Details

### 1. Service worker fetch bypass + cache bump

**File**: `public/sw.js`

**Current state**:
- `const CACHE_NAME = 'brew-log-v1'`; `install` pre-caches `SHELL` + `skipWaiting`;
  `activate` deletes caches `!== CACHE_NAME` + `clients.claim`.
- `fetch` handler: `return`s for non-GET; navigation requests are network-first
  with an `index.html` fallback; **all other GETs are cache-first** (`caches.match`
  → cached, else fetch and `cache.put` when `r.ok`). No origin filtering — this is
  the line that caches cross-origin Supabase responses.

**Modifications**:
- Change `CACHE_NAME` to `'brew-log-v2'` so `activate` purges the poisoned v1
  cache and the new SW rolls out.
- In the `fetch` handler, immediately after the existing non-GET guard, parse the
  request URL and **`return` early for Supabase hosts** (host ending in
  `.supabase.co`). This must come **before** the navigation branch and the
  cache-first branch, so no Supabase request is ever matched against or written
  to the cache.
- Leave the navigation (network-first) and same-origin/other cache-first branches
  exactly as they are.

**Why host-match (not env injection)**: `public/sw.js` is copied verbatim by Vite
(not transformed), so `VITE_SUPABASE_URL` is not available inside it. The project's
Supabase lives on the default `*.supabase.co` domain, so matching that host is
correct and self-contained. (Assumption noted: if a custom Supabase domain is ever
adopted, this check must be updated — called out in Edge Cases.)

### 2. Resync when the new SW takes control (rollout correctness)

**File**: `src/main.jsx`

**Current state** (lines 17–33): in production it registers `/sw.js` only inside a
`window.addEventListener('load', …)` callback. There is **no `controllerchange`
handling**. The app's first sync starts on mount (BrewContext cache-open effect),
which runs **before** `load`.

**The rollout gap** (Codex P1): on the first visit after deploy, the page is still
controlled by the **old v1 SW**. The app mounts and `pullRemote` runs — served the
**stale 22-row** response by v1 — *before* `load` fires and registration even
begins. The new v2 SW then installs (`skipWaiting`) and `clients.claim()`s, but
that claim does **not** replay the already-completed stale pull. Result: the user
still sees stale data until they manually reload a **second** time — precisely the
"I refreshed and it didn't work" trap.

**Modifications**:
- Add a **guarded `controllerchange` listener** (production only): when a new
  service worker takes control of the page, **trigger one resync** rather than a
  full page reload. Once `controllerchange` has fired, `navigator.serviceWorker.controller`
  is the new v2 worker, so *every* subsequent fetch from the page — including the
  resync's `pullRemote` — goes through v2 (which bypasses the SW cache for Supabase)
  and hits the network for fresh rows. Implement the resync by reusing the existing
  wake path: dispatch the same event `BrewContext` already re-syncs on
  (`window` `online`/`focus` → `onWake` → `runSync`), so no new coupling or context
  plumbing is needed. A boolean guard makes it fire at most once.
- Keep the existing `register('/sw.js')` on `load` and the dev-mode unregister
  branch unchanged.

**Note**: a resync — **not** `location.reload()` — is chosen deliberately. A full
reload would discard an in-progress **add** or **edit** draft, which lives only in
React state (`AddPage`, `DetailPage`) and is not yet persisted. Because v2 already
controls the page after `controllerchange`, a resync alone refreshes entries and
photos over the network with no reload and no lost input. (App-shell/code updates
still arrive on the next natural navigation, as with the existing SW model; this
handler's job is only to un-stale the data.) On a first-ever install (no prior
controller) the resync is a harmless no-op duplicate of the mount sync.

## Technical Considerations

- **Pattern Usage**: keeps the existing SW structure (install/activate/fetch,
  cache-name-scoped cleanup); only adds an origin guard and bumps the version.
- **Offline-First Impact**: none to the data model. Offline reads of entries and
  photos come from IndexedDB, unaffected. The SW change only stops the SW from
  short-circuiting Supabase network reads with stale cached copies.
- **Sync Safety**: no change to `sync.js`, the outbox, `pullRemote`/`flushOutbox`,
  or per-user cache isolation. This fix makes `pullRemote`'s network read actually
  reach the server instead of a stale SW cache — it *restores* intended sync
  behavior rather than altering it.
- **Mobile/PWA UX**: the updated SW must be fetched by each device to take effect.
  Browsers check for SW updates on navigation (~every 24h); an installed iOS PWA
  may need to be fully closed and reopened (or the app re-added) to pick it up. The
  DevTools "Bypass for network" toggle is the interim workaround until the new SW
  activates. Once active, `skipWaiting` + `clients.claim` (already present) make it
  control open clients immediately, the new `controllerchange` handler in
  `main.jsx` triggers a one-time resync so the fresh pull actually runs under v2
  (closing the Codex P1 rollout gap) without reloading the page, and the v2 cache
  bump purges the old poison.
- **Theming**: unaffected — Google Fonts remain SW-cached (cache-first) per the
  "bypass Supabase only" decision, so offline theme fonts still work.
- **Performance**: negligible; Supabase reads already went over the network on a
  cache miss. Removing the (incorrect) cache hit for these requests only ensures
  freshness. App-shell offline start is unchanged.
- **Edge Cases**:
  - Custom Supabase domain (not `*.supabase.co`) would bypass the guard — documented
    assumption; update the host check if that ever changes.
  - Devices on the old SW keep showing stale data until the new SW activates — the
    cache bump + normal SW update flow resolves this; interim workaround is
    "Bypass for network" or clearing the site's Cache Storage.
  - Same-origin `/auth` callback and static assets are unaffected (not Supabase-host).

## Type Definitions

None.

## Performance & Cost Impact

- No cost change. Reads that were (incorrectly) served from the SW cache now hit
  Supabase over the network — the intended behavior; response sizes are unchanged.

## Backward Compatibility

- Fully backward compatible: no schema, DB, or API changes. The `CACHE_NAME` bump
  purges the old HTTP cache on the client; IndexedDB data and login are untouched.

## Test Impact

- **No automated tests**: `sw.js` and `main.jsx`'s SW-registration block have no
  harness (ARCHI §10 — service worker not covered; `sw.js` is a plain `public/`
  script not imported by Vitest, and the registration code is browser-only). The
  changes are a one-line origin guard, a constant bump, and a small
  `controllerchange` handler; wiring these into Vitest would cost more than it
  verifies.
- **Manual verification (acceptance)**: after deploy, on a device that was
  previously showing stale data, load the app **with "Bypass for network" off**.
  Concrete, verifiable signals (the SW script URL is unchanged, so
  `navigator.serviceWorker.controller` can't reveal the version — use these instead):
  1. **Cache Storage** (DevTools → Application → Cache Storage) shows **`brew-log-v2`**
     and **no** `brew-log-v1` — proof the new SW installed/activated and purged the
     poison.
  2. Re-run the DIAG snippet: `pullCalls` report the **current** server row count
     (e.g. 40) and newest date, not the stale 22/June-20.
  3. The newest brews appear in the feed without the DevTools bypass (the resync
     landed them).
  Also confirm the app still cold-starts offline (app shell from the v2 cache) and
  theme fonts still load, and that being mid-add/edit is **not** interrupted (no
  reload).
- **Existing suite**: unaffected — no `src/**` logic changes; `npm test` should
  stay green.
