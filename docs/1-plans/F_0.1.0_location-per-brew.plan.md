# Location per Brew Implementation Plan

## Overview

Add an optional **location** to each brew entry: a free-text label ("home", "Blue Tokai Khan Market") typed on the Add form, shown on the feed card and detail page, and editable after the fact. The field syncs to Supabase like every other entry field. The data shape and UI seam are deliberately structured so a later Google Maps / OpenStreetMap picker can add structured data (coordinates, place id) without a breaking change.

## Problem Statement

Entries currently capture what was brewed and when, but not where. For a log that mixes home brews and caf├® visits, "where" is a natural axis (and a prerequisite for any future map view). There is also no edit path for existing entries at all ŌĆö DetailPage is read-only ŌĆö so adding an editable field requires the first `updateEntry` mutation.

## Solution Architecture

- **Client entry shape**: add optional `location: string` (present only when set, following the existing `rating`/`notes`/`color` conditional-spread pattern). `isValidEntry` is untouched ŌĆö it only requires `id`/`name`/`timestamp`, so old cached entries keep loading.
- **Reserved for maps (not built now)**: a future optional `locationMeta` object (`{ lat, lng, placeId, provider }`) on the client and a `location_meta jsonb` column in Supabase. Nothing is added today; this plan just fixes the names so the map feature is purely additive.
- **Sync**: `location` maps to a new nullable `location` text column on the `entries` table, handled symmetrically in `toRow`/`fromRow`.
- **Edit path**: a new `updateEntry(id, patch)` in BrewContext merges the patch into the cached entry and enqueues a **new `update` outbox op** carrying only `{ id, patch }`. `flushOutbox` replays it as a partial row update (`.update(patchRow).eq('id', id)`) that **never touches Storage and never writes `photo_path`**. Reusing the `add` op was considered and rejected: the Storage policies grant no `UPDATE` (`docs/superpowers/supabase-setup.md`), so re-uploading an already-synced photo with `upsert: true` would be rejected and stall the outbox ŌĆö and replaying a stale `photo_path: null` could clobber the remote path. The ordered outbox guarantees the `add` op lands before any `update` for the same entry.
- **Input UX**: a reusable `LocationField` component ŌĆö text input plus tappable chips of recently used locations derived from cached entries. This component is the seam where a map-picker button lands later.

**Modifications**:

## Technical Considerations

- **Pattern Usage**: conditional-spread optional fields (matches `rating`/`notes`/`color`); updates flow through the same ordered outbox as a new `update` op replayed as a whitelisted partial row update; pure helpers in `src/utils/`.
- **Offline-First Impact**: `location` is optional everywhere; `isValidEntry` unchanged; old cached entries and old backups load unmodified. `toRow`/`fromRow` stay symmetric.
- **Sync Safety**: `updateEntry` enqueues through the same ordered outbox, so addŌåÆupdateŌåÆdelete replay order is preserved. The `update` op is a text-only partial row update ŌĆö no Storage calls (Storage policies grant no `UPDATE`), no `photo_path` writes. No cache DB version bump (no new store, entries store is schemaless; unknown `update` ops are simply never produced by old code). Per-user isolation untouched. Known inherited behavior: if a flush fails and `pullRemote` runs, a pending local update can be overwritten by the remote row ŌĆö this is the app's existing conflict semantic for the outbox, not a regression introduced here.
- **Mobile/PWA UX**: chips sized for touch; input uses existing sheet styles; works fully offline (suggestions come from cache, edit queues in outbox).
- **Theming**: all new UI uses theme CSS variables; check chips/labels across all five themes.
- **Performance**: `recentLocations` is O(n) over in-memory entries, memoized; no new re-render sources in BrewContext beyond the state update `updateEntry` already requires.
- **Edge Cases**: entries with no location (feed/detail render nothing extra); clearing a location on edit (field removed, syncs `null`); offline update while signed out is impossible (app requires sign-in for cache); duplicate-ish suggestions ("Home" vs "home") deduped case-insensitively; very long labels truncated in the card.

## Type Definitions

Client entry shape gains one optional field:

- `location?: string` ŌĆö free-text place label; absent when unset.

## Backward Compatibility

- Old cached entries / backups lack `location` ŌåÆ load unchanged (`isValidEntry` untouched, field optional).
- Old Supabase rows have `location = null` ŌåÆ `fromRow` omits the field.
- Column added before client ships ŌåÆ no outbox stalls; old installed PWAs simply never write the column.
