# Detail-Page Colour Picker Implementation Plan

## Overview

Add drink-colour editing to the brew **Detail page** by reusing the existing pick-from-photo `ColorPicker`. Today the colour picker only lives on the Add page, so once a brew is logged (including every seeded demo brew) there is no way to re-pick its colour. This plan makes `color` an inline-editable field on `DetailPage`, wired through the same `updateEntry` → outbox `update` path as the other editable fields, so it works identically in demo mode (local-only) and real mode (synced).

## Problem Statement

- `ColorPicker` is mounted only in `AddPage` (`src/pages/AddPage.jsx:99`). It samples a colour by touching an uploaded photo.
- `DetailPage` supports inline editing of name, type, date, rating, notes, and location — but **not colour**.
- Demo brews (`src/lib/demoData.js`) all ship with photos and a seeded `color`, but the demo user (the primary place someone explores editing) can never re-pick that colour, because the only picker is on the Add page.
- The `update` outbox op whitelist (`normalizePatch`/`toPatchRow`/`applyPatch` in `src/lib/sync.js`) does not currently carry `color`, so even if the UI existed, edits wouldn't persist.

## Solution Architecture

Two coordinated changes:

1. **Sync layer** — extend the canonical `update` patch to include `color` (a `#rrggbb` hex string), symmetric across `normalizePatch` → `toPatchRow` → `applyPatch`. The Supabase `entries.color` column already exists (`toRow`/`fromRow` map it today), so **no migration is required** and the change is backward compatible with cached rows.
2. **UI layer** — add a colour row to `DetailPage` that renders a swatch + "Pick from photo" affordance, opening the existing `ColorPicker` in an edit sub-state with Save/Cancel, calling `updateEntry(id, { color })`. `ColorPicker` gains a small opt-out so it does **not** auto-overwrite the existing colour on mount.

Because `DetailPage` and the sync path are shared, this feature lights up in **both** demo and real mode from a single implementation. Demo mode simply writes to its isolated cache and never flushes (per the demo sync gate), exactly as the other editable fields already behave.

## Implementation Details

### 1. Extend the update patch to carry `color`

**File**: `src/lib/sync.js`

**Current state**: `normalizePatch` (lines 44–73), `toPatchRow` (79–100), and `applyPatch` (106–130) handle a fixed set of fields — `name`, `type`, `rating`, `notes`, `location`, `timestamp`. There is no `color` branch.

**Modifications**:

- **`normalizePatch`** — add a `color` branch: when the patch has own-property `color`, coerce to a trimmed string and accept only a valid 6-digit hex (`/^#[0-9a-fA-F]{6}$/`, case-insensitive). Invalid/empty values are dropped (the field is never cleared to `null` from this UI, since a colour is always present once a photo exists; treat non-matching input as "no change"). Rationale: every colour the app produces — `extractDominantColor`, `sampleCanvasAt`, and seeded demo specs — is `#rrggbb`, so strict hex validation is safe and guards the synced column.
- **`toPatchRow`** — when the normalized patch has `color`, set `row.color = patch.color`. Mirrors the existing column mapping in `toRow` (line 14).
- **`applyPatch`** — when the patch has `color`, set `next.color = patch.color` (colour is always a present value here, so no delete branch needed, unlike the optional `rating`/`notes`/`location` clear cases).

### 2. Make `ColorPicker` reusable without clobbering an existing colour

**File**: `src/components/ColorPicker.jsx`

**Current state**: The mount effect (lines 11–14) calls `extractDominantColor(photoDataUrl).then(onChange)` whenever `photoDataUrl` changes. On the Add page this is desired — a freshly attached photo seeds the initial colour. On the Detail page it is **harmful**: mounting the picker over an existing brew would immediately replace the saved colour with the auto-extracted dominant colour.

**Modifications**:

- Add a prop `autoExtract` (default `true`) to the component signature.
- Guard the mount effect so it only runs `extractDominantColor(...)` when `autoExtract` is true. When false, the component still renders the swatch + "Pick from photo" flow and only calls `onChange` on an explicit user sample (`handlePointerUp`).
- No change to the existing Add-page call site behaviour (default keeps `autoExtract` true).
- The component already returns `null` when there is no `photoDataUrl` (line 60), which is the correct behaviour for photoless entries on the Detail page (see Edge Cases). It also already uses theme variables for all colours.

### 3. Add the colour edit row to the Detail page

**File**: `src/pages/DetailPage.jsx`

**Current state**: Inline editing follows a consistent pattern: a `editingField` state string, a per-field draft state, `begin<Field>Edit()` / `handle<Field>Save()` handlers, and a rendered row that toggles between a display button and an edit block with `<EditActions>`. Colour is absent. The component already has access to `entry.photo` (a same-origin blob object URL used by the hero `<img>`, lines 161–162) and `entry.color`.

**Modifications**:

- Import `ColorPicker` (`../components/ColorPicker`).
- Add draft state `colorDraft` and include `'color'` as an `editingField` value.
- Add `beginColorEdit()` — set `colorDraft` from `entry.color` (fallback to the app's default swatch used elsewhere, `#c97b3a`) and set `editingField` to `'color'`.
- Add `handleColorSave()` — `await updateEntry(id, { color: colorDraft })` then clear `editingField`. Cancel restores nothing (draft is discarded on next open).
- Render a colour row alongside the other rows (place it near the rating/notes rows for a natural grouping). Two states:
  - **Display state**: a button showing a colour swatch (circle filled with `entry.color || '#c97b3a'` — the neutral default, so backward-compatible colourless entries do not render a transparent swatch) + label ("Drink colour"), styled with theme variables, consistent with the other `detail-*-row` buttons.
  - **Edit state**: a `detail-edit-row` containing `<ColorPicker photoDataUrl={entry.photo} color={colorDraft} onChange={setColorDraft} autoExtract={false} />` plus `<EditActions onSave={handleColorSave} onCancel={() => setEditingField(null)} />`.
- **Photoless / photo-unavailable entries**: gate the pick-from-photo edit affordance on **`entry.photo`** (a usable object URL) — **not** `hasPhoto`. `hasPhoto` can remain `true` when the local blob download failed, which would leave `ColorPicker` with no usable image URL (it renders `null`). When `entry.photo` is falsy, render the swatch as a non-interactive display and omit the "pick" affordance rather than opening an empty picker. Demo brews always have a loaded photo, so the demo path is fully exercised.

### 4. Styling

**File**: `src/index.css`

**Modifications**: Add any small class(es) needed for the colour display row/swatch (e.g. `detail-color-row`), reusing existing `detail-*-row` conventions and **theme CSS variables only** (`--surface`, `--border`, `--border-strong`, `--text-secondary`, `--radius-sm`, etc.). No hardcoded colours except the neutral `#c97b3a` default already used across the codebase as the colour fallback. Verify appearance across all five themes.

## Technical Considerations

- **Pattern Usage**: Follows the established `DetailPage` inline-edit pattern (draft state + `begin`/`handle` + `<EditActions>`) and the `normalizePatch`/`toPatchRow`/`applyPatch` symmetry from ARCHI §5.
- **Offline-First Impact**: `color` already exists in the client entry shape and in `isValidEntry`-compatible cached rows; no schema addition. The change only makes an existing field editable. Old cached entries without a `color` still load; editing simply sets one.
- **Sync Safety**: `color` joins the canonical patch, replayed in `seq` order like every other `update` field. No new op type, no ordering change, no photo-storage interaction (`color` never touches `photo_path` or blob upload). Per-user cache isolation and demo's no-sync gate are unchanged — demo edits stay in the demo cache and are cleared on reseed.
- **Mobile/PWA UX**: Reuses the existing touch-sampling picker (already `touch-action: none`, magnifier, large touch targets). Works offline (local cache write; sync deferred). Detail-page edit blocks already respect safe-area layout.
- **Theming**: New row/swatch uses theme variables only; verified across all five themes. `ColorPicker` already theme-compliant.
- **Performance**: `buildPickerCanvas` runs only when the user taps "Pick from photo", not on row render, so there is no cost for merely viewing a brew. Disabling `autoExtract` on the Detail page also avoids an unnecessary `extractDominantColor` pass on mount.
- **Edge Cases**: (a) Entry without a photo → no pick-from-photo affordance (swatch shown read-only). (b) `autoExtract=false` prevents the mount effect from overwriting the saved colour. (c) Invalid/duplicate colour strings dropped by `normalizePatch` hex validation. (d) A concurrent remote delete still wins (delete-wins) — a colour `update` for a deleted row matches zero rows and is pruned, same as other fields. (e) Cancel discards `colorDraft` without a write.

## Type Definitions

No shape changes. `color: string|null` already exists in the client entry (ARCHI §6) and the Supabase `entries.color` column. This plan only widens the set of fields the `update` op may carry.

## Backward Compatibility

Fully backward compatible. `color` is an existing optional field; the Supabase column already exists (no migration). Cached entries without `color` continue to load and can now receive one via editing. Older app versions ignore an unknown-to-them nothing — there is no new column or op type. Demo mode is unaffected structurally (reseed still clears the demo outbox).

## Test Impact

- **`src/lib/sync.test.js`** (existing coverage of `normalizePatch`/`toPatchRow`/`applyPatch`): add cases for `color` — valid hex accepted and mapped symmetrically; invalid/non-hex dropped; round-trip `normalizePatch → applyPatch` preserves the colour; `toPatchRow` sets the `color` column only when present.
- **`isValidEntry` / cache tests**: no new field, so no change expected; confirm an entry edited to a new colour still validates.
- **Components (`ColorPicker`, `DetailPage`)**: not covered by the Vitest harness (no jsdom/RTL). The `autoExtract` guard and the Detail-page row are verified manually. **Demo mode** (tap a seeded brew → Pick from photo → Save): the new colour persists across in-session navigation (leave and re-open the brew), and is expected to **reset to the seeded colour on reload** — demo reseeds its cache and clears the demo outbox on open, so reload persistence is *not* an acceptance criterion for demo. **Real (signed-in) mode** is where the colour must persist across reload (cache write + outbox flush). The pure sync-layer changes carry the automated coverage.
- No integration/E2E harness exists; manual demo-mode walkthrough is the acceptance check.
