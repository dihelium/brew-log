# Demo Mode (No-Sign-In Reviewer Experience) Implementation Plan

## Overview

Add a **demo mode** so anyone reviewing the app can explore it without a Google
sign-in. From the login screen a reviewer taps "Explore the demo," which drops
them into a local, pre-seeded brew log running entirely off IndexedDB вЂ” no
Supabase auth, no network writes. The demo mug is populated with three brews
(two coffees + one matcha, favoring photos with a laptop/desk background),
re-dated to "today" on every entry so the daily-cup visualization is always
full and current.

## Problem Statement

Today `App.jsx`'s `Gate` renders `LoginPage` for any signed-out visitor, and
`LoginPage` offers only "Continue with Google." A reviewer with no Google
account (or who doesn't want to hand over their identity) cannot see the app at
all. There is no way to experience the feed, daily cup, calendar, or editing
flows without authenticating against the real backend.

## Solution Architecture

Introduce a synthetic, local-only demo session that reuses the entire existing
offline-first stack (BrewProvider, cache, pages) while **disabling the sync
path**:

1. **Demo session in `AuthContext`** вЂ” new `isDemo` state plus `enterDemo()` /
   `exitDemo()`. When demo is active and no real Supabase user exists, the
   context exposes a synthetic user `{ id: 'demo', email: 'demo@brew.log',
   isDemo: true }`. `Gate` already renders the app whenever `user` is truthy, so
   no `Gate` structural change is required beyond passing the demo flag down.
   A `localStorage['brewlog:demo']` flag keeps demo mode across PWA reloads.

2. **Local-only sync gating in `BrewContext`** вЂ” `runSync` becomes a no-op when
   `isDemo` is true. This is the critical sync-safety guarantee: `flushOutbox`
   never pushes demo rows to Supabase and, crucially, `pullRemote` never runs
   (it would otherwise prune the local demo entries as "not on the server").
   Add/edit/delete still write to the local cache and update React state, so the
   demo is fully interactive вЂ” the mutations simply never leave the device.

3. **Fresh reseed on entry** вЂ” when the provider opens the `brew-log-demo`
   cache in demo mode, it clears the three stores and re-seeds the three brews,
   re-dated to the current day. This is done via pure, node-testable seed
   builders in `src/lib/`, with the (large, base64) photo payloads isolated in a
   dynamically-imported module so they never weigh down the main bundle.

4. **Demo affordance in the UI** вЂ” a "Explore the demo" button on `LoginPage`
   and a small "Demo mode В· Sign in" chip on `FeedPage` that calls `exitDemo()`
   to return to the login screen.

No entry-schema change, no `toRow`/`fromRow` change, no IndexedDB version bump,
no Supabase migration. The demo entry rows use the exact same cached-row shape
that `importEntries` already produces.

## Implementation Details

### 1. Demo seed data (photos + specs)

**File**: `src/lib/demoData.js` (new)

Holds the three static brew specs including their photo data-URLs, extracted
from `brew-log-backup-2026-06-20.json`:

| Slot | Source entry | type | name | color | rating | laptop? |
| --- | --- | --- | --- | --- | --- | --- |
| 0 | Orange Spiced Cappuccino (idx 0) | `coffee` | Orange Spiced Cappuccino | `#c97b3a` | 4 | no |
| 1 | Vanilla Latte (idx 1) | `coffee` | Vanilla Latte | `#c97b3a` | 5 | yes |
| 2 | Hot Matcha Latte (idx 7) | `matcha` | Hot Matcha Latte | `#808751` | 4 | yes |

- Slot 2's source is typed `coffee` in the backup (all 22 entries are); the
  spec **overrides `type` to `matcha`** to satisfy the "one matcha" requirement.
- Each spec: `{ id, type, name, color, rating, photo }` where `photo` is the
  `data:image/jpeg;base64,...` string and `id` is a stable demo id
  (`'demo-brew-0/1/2'`, deliberately not the real UUIDs).
- This module is **dynamically imported** (`await import('./demoData.js')`) only
  when seeding, so the ~370 KB of base64 lands in its own lazy chunk.
- **Offline behavior**: `public/sw.js` runtime-caches any fetched GET asset that
  returns `ok` (its `fetch` handler, not just the install-time `SHELL`), so once
  a reviewer has entered the demo **once while online**, the chunk is cached and
  the demo works offline thereafter. The remaining gap is the cold case вЂ” an
  installed PWA, offline, entering demo for the very first time: the dynamic
  import fails. That is handled by seed-failure UX (see В§6), **not** by
  pre-caching the chunk at install (which would push 370 KB onto every user,
  including those who never open the demo).

**Why a build script extracts the photos:** rather than hand-pasting base64, add
a one-off dev script `scripts/extract-demo-photos.mjs` (new) that reads the
backup JSON and writes `src/lib/demoData.js`. This keeps provenance clear and
lets the seed be regenerated. The script is not part of the runtime/build.

- **Input**: the script takes the backup JSON path as a **required CLI
  argument** (it is not committed to the repo). Documented invocation:
  `node scripts/extract-demo-photos.mjs <path-to-backup.json>` вЂ” for this
  feature the source is
  `/Users/bhargavvarshney/Downloads/brew-log-backup-2026-06-20.json`. The script
  exits with an error if the path is missing or unreadable. Only its committed
  output (`src/lib/demoData.js`) is required at runtime; the backup file itself
  is never needed after generation.

### 2. Pure seed builders

**File**: `src/lib/demo.js` (new)

Pure, node-testable logic (keeps browser APIs out for Vitest):

- `demoTimestamps(now)` вЂ” given a `Date`/ms `now`, returns three ascending ms
  timestamps on the **same calendar day** as `now`, all **at or before `now`**
  so no brew is future-dated (a morning reviewer must not see a 16:45 brew).
  Approach: anchor the newest at `now`, step back in fixed increments (e.g.
  `now`, `now в€’ 2h`, `now в€’ 4h`), and clamp each to `в‰Ґ` start-of-day so an
  early-morning `now` still yields three same-day times (bunched near midnight
  if necessary). All three pass `FeedPage`'s `toDateString()` "today" filter and
  the daily cup shows three ordered bands.
- `buildDemoEntries(specs, now)` вЂ” maps the imported specs to cached-row
  objects: `{ id, type, name, timestamp, photo_path: null, hasPhoto: true,
  rating, color }`, timestamps from `demoTimestamps`. Ordered oldestв†’newest so
  the mug bands read correctly (bottom = oldest). Returns rows **plus** each
  row's photo data-URL for blob storage.

**Note**: `buildDemoEntries` returns plain data only; it does not touch the
cache or `crypto`/`URL`, so it is unit-testable in the node environment.

### 3. Cache: add a `clear()` accessor

**File**: `src/lib/cache.js` (modify)

**Current state**: `createCache` returns accessors for entries/photos/outbox but
no way to wipe them.

**Modifications**:
- Add `clear: () => Promise<void>` to the returned accessor object that clears
  all three stores (`entries`, `photos`, `outbox`) вЂ” used to guarantee a
  pristine reseed. Implement via `db.clear(store)` for each store.
- **No DB version bump** вЂ” this is a data operation, not a schema change; the
  `openDB(..., 1, ...)` version stays `1`.

### 4. Demo seeding orchestration

**File**: `src/lib/demo.js` (same new module, impure helper)

- `seedDemoCache(cache, now)` вЂ” `await`s the dynamic import of `demoData.js`,
  calls `buildDemoEntries`, then: `await cache.clear()`, and for each row
  `await cache.putPhotoBlob(id, dataUrlToBlob(photo))` + `await
  cache.putEntry(row)`. It **does not** call `cache.enqueue` вЂ” demo rows must
  never enter the outbox. Uses `dataUrlToBlob` from `photoCodec.js`.

### 5. AuthContext: demo session

**File**: `src/context/AuthContext.jsx` (modify)

**Current state**: tracks `user`/`loading` from the Supabase session; exposes
`signInWithGoogle`, `signOut`.

**Modifications**:
- Add `isDemo` state, initialized from `localStorage['brewlog:demo'] === '1'`.
- Derive the exposed user: `const effectiveUser = realUser ?? (isDemo ? DEMO_USER
  : null)` where `DEMO_USER = { id: 'demo', email: 'demo@brew.log', isDemo: true }`.
  A real Supabase session always wins over demo.
- `enterDemo()` вЂ” set `isDemo = true`, persist the flag. `exitDemo()` вЂ” set
  `isDemo = false`, remove the flag.
- `signOut()` вЂ” also call `exitDemo()` so signing out never leaves a stray demo
  flag.
- If `onAuthStateChange` yields a real session while `isDemo` is set, clear the
  demo flag (real user takes over).
- Provide `isDemo`, `enterDemo`, `exitDemo` through the context value.

### 6. BrewContext: local-only demo, seed on open

**File**: `src/context/BrewContext.jsx` (modify)

**Current state**: opens `brew-log-<userId>` on `userId` change, hydrates, then
`runSync()` (flush + pull). `runSync` also fires on `online`/`focus`.

**Modifications**:
- Read `isDemo` from `useAuth()`.
- **Gate `runSync`**: early-return when `isDemo` (in addition to the existing
  `!cache || !userId` guards). This alone prevents `flushOutbox` and
  `pullRemote` from ever running in demo вЂ” mutations stay in cache/state.
- **Cache-open effect**: when `isDemo` and `userId === 'demo'`, after
  `createCache`, `await seedDemoCache(cache, Date.now())` **before** `hydrate`,
  and skip the trailing `runSync()`. For real users the path is unchanged.
  Wrap the seed in `try/catch`: if it throws (cold-offline first entry, where
  the `demoData.js` dynamic import can't be fetched), set a `demoSeedError`
  state so the UI can show a retry/"connect to load the demo" message instead of
  a silently empty feed.
- **Retry data flow**: `FeedPage`'s retry action calls `window.location.reload()`
  rather than an in-page reseed. Reason: the only realistic seed failure is the
  `demoData.js` dynamic import failing while offline, and a **failed module
  fetch is cached in the document's module map** вЂ” a same-document re-`import`
  returns the cached rejection and can never recover. Reloading re-enters demo
  (the `brewlog:demo` localStorage flag persists) and refetches `demoData.js`
  fresh once the reviewer is back online; if still offline, the same
  `demoSeedError` state is shown again. `BrewContext` exposes `demoSeedError`;
  no in-context retry function is needed.
- **Mutations and the demo outbox**: add/edit/delete already `enqueue` to the
  outbox and then call `runSync()`. In demo, `runSync` no-ops, so these ops
  accumulate in the **isolated `brew-log-demo` outbox and are never flushed to
  Supabase**; they are wiped on the next reseed. This is intentional вЂ” no
  branch in the mutation methods is needed, and the ops can never reach the real
  backend because `flushOutbox` never runs in demo. (Only `seedDemoCache` is
  explicitly enqueue-free, so the seeded rows themselves never enter the
  outbox.)

### 8. FeedPage: demo chip + exit

**File**: `src/pages/FeedPage.jsx` (modify) and `src/components/DemoBanner.jsx`
(new)

**Modifications**:
- New `DemoBanner` component: a slim, theme-variable-styled chip reading
  "рџ§Є Demo mode В· Sign in" that calls `exitDemo()` (from `useAuth`) on tap,
  returning the reviewer to `LoginPage`. Non-fixed, flows in the layout so it

## Technical Considerations

- **Pattern Usage**: Reuses the existing offline-first stack unchanged вЂ” demo is
  just a per-user cache (`brew-log-demo`) with sync switched off. Seed rows use
  the same shape `importEntries` builds; `isValidEntry` passes.
- **Offline-First Impact**: No entry-schema change, no new optional fields, no
  `toRow`/`fromRow` change. Old cached entries for real users keep loading
  untouched. The demo cache is a separate IndexedDB database.
- **Sync Safety**: The core guarantee is `runSync` no-ops in demo, so
  `flushOutbox`/`pullRemote` never run вЂ” no writes to the real Supabase backend
  and no remote-delete pruning of demo entries. Seeded rows never enter the
  outbox (`seedDemoCache` skips `enqueue`); reviewer mutations may enqueue into
  the isolated `brew-log-demo` outbox, but it is never flushed and is wiped on
  reseed, so nothing reaches Supabase. Per-user isolation is preserved: real
  user ids are UUIDs and can never collide with the literal `'demo'`.
- **Mobile/PWA UX**: Demo button and chip are в‰Ґ44px touch targets, theme-styled,
  safe-area friendly. Demo works fully offline; the lazy `demoData.js` chunk is
  cached by the service worker after first load.
- **Theming**: All new UI (`LoginPage` button, `DemoBanner`) uses theme CSS
  variables; verified against all five themes вЂ” no hardcoded colors.
- **Performance**: Photo data-URLs (~370 KB base64) live in a dynamically
  imported chunk, so they are excluded from the main bundle and only fetched
  when a reviewer enters the demo. Photos are already compressed in the backup.
  No new BrewContext re-render paths for real users.
- **Edge Cases**: Reseed each entry guarantees a full, today-dated mug even
  across a day boundary or PWA reload (localStorage flag re-enters demo and
  reseeds). Real session appearing during demo clears the demo flag. Signing out
  clears the demo flag.

## Type Definitions

No new persisted types. In-memory only:
- Synthetic user: `{ id: 'demo', email: 'demo@brew.log', isDemo: true }`.
- Demo spec (module-internal): `{ id, type, name, color, rating, photo }`.
- Seeded cached row: identical to the existing cached-row shape produced by
  `importEntries` вЂ” `{ id, type, name, timestamp, photo_path: null, hasPhoto,
  rating, color }`.

## Performance & Cost Impact

- Main bundle unaffected (photos code-split into a lazy chunk).
- Zero backend cost: demo never touches Supabase (no auth, no reads/writes, no
  storage). This is also the point вЂ” reviewers cannot pollute real data.

## Backward Compatibility

- Fully backward compatible. No schema, DB version, or sync-mapping changes.
- Real (Google) users are unaffected; the demo path is only reachable while
  signed out and only when `enterDemo()` is invoked.

  correct colors вЂ” all node-testable (no browser APIs). Include a **midday case**

- **Existing tests**: `sync.js`, `photoCodec.js`, `theme.js`, `streakCalc.js`

- **Not unit-tested (manual/browser verification)**: `runSync` demo gating,

  React layer with no test harness (per ARCHI В§10). Verify manually: enter demo
