# Changelog - Week 1, 15-07-2026, V. 0.1.0

**Release Date**: Week 1, 15-07-2026 at 08:34

**Code review**: `docs/3-code-review/CR_w1_v0.1.0.md` (Codex loop, 2 rounds -> APPROVED with observations)

## Changes

- **Location field on entries**: optional free-text `location` on the client entry shape, mapped symmetrically in `toRow`/`fromRow` to a new nullable `location` text column on the Supabase `entries` table. Old cached entries and backups load unchanged.
- **First edit path**: new `updateEntry(id, patch)` in `BrewContext`, replayed through a new `update` outbox op as a whitelisted partial row update (`toPatchRow`) that never touches Storage — Storage policies grant no `UPDATE`, so photo re-upload was deliberately avoided.
- **Sync race hardening** (from code review): `runSync` queues a follow-up pass when called mid-sync, and `pullRemote` overlays pending update patches (`applyPatch`) so an edit enqueued during an active sync is not reverted by the stale remote row.
- **UI**: `LocationField` component (text input + recent-location suggestion chips, future map-picker seam) on AddPage; location shown on EntryCard meta row and DetailPage with inline edit; theme-variable styles only.
- **Helpers/tests**: `recentLocations` pure helper (case-insensitive dedupe, newest-first); 13 new Vitest cases covering mapping, patch whitelisting, update replay, overlay behavior, and suggestions.
- **Backups**: `exportEntries`/`importEntries` round-trip the location field.

## Migration

Applied to Supabase before deploy (column-first order, safe for old clients):
