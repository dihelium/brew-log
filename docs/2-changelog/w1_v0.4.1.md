## Changes

Fixes a confirmed production bug where devices showed stale data indefinitely even though the server had newer brews.

- **Root cause**: `public/sw.js` used a cache-first strategy for *all* GET requests, with no origin check, so it cached the cross-origin Supabase `GET /rest/v1/entries` response (when the server had 22 rows) and served that stale response forever. Writes are POST/PATCH and passed through, so data reached the server (40 rows) but no device could read the update. Confirmed via device diagnostics: the app's pull returned 22 rows/June while the DB held 40, and DevTools "Bypass for network" immediately showed all 40.
- **Fix (`public/sw.js`)**: the service worker now **bypasses its cache for `*.supabase.co` requests** (always network, so reads are fresh), while same-origin app assets and Google Fonts keep their cache-first behavior. `CACHE_NAME` bumped `brew-log-v1` → `brew-log-v2` so the `activate` handler purges the poisoned cache.
- **Rollout (`src/main.jsx`)**: a guarded `controllerchange` listener triggers **one resync** (via the existing `online` wake path) when the updated worker takes control, so fresh data loads under the new worker immediately — without a page reload that would discard an in-progress add/edit draft.

No entry-schema, IndexedDB, sync-logic, or Supabase changes. Offline still works (entries/photos come from IndexedDB, not the SW HTTP cache).
